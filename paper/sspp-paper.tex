\title[VORTEX-HRM]{VORTEX-HRM: Hierarchical Retrieval-Augmented Generation for Multi-Hop Question Answering}

\author[V.~Malykh]{Valentin Malykh}
\address[V.~Malykh]{Skolkovo Institute of Science and Technology, Moscow, Russia}
\email{valentin.malykh@skoltech.ru}

\author[D. Zelenin]{Dmitrii Zelenin}
\address[D. Zelenin]{GitHub: \url{github.com/dizel0110/Text-HRM-RAG}}
\email{}

\begin{abstract}
We present VORTEX-HRM, an agentic retrieval-augmented generation system for multi-hop question answering. Unlike prior work that relies on LLM-driven tool selection or flat ReAct loops, VORTEX separates planning from execution using a fact-free planner and a fixed chained retrieval pipeline. The Gravitational Core (planner) holds zero factual knowledge in its weights---it is a pure structural router communicating via an XML-based protocol. The Centrifugal Ingestor (executor) resolves sub-questions through a deterministic pipeline: keyword search, chunk reading, and LLM condensation. The cycle repeats---each rotation is a spiral---until convergence via entropic collapse. On a 50-question multi-domain benchmark with three LLMs, VORTEX achieves 74\% Contains accuracy (qwen2.5:7b, T4 GPU), outperforming Naive RAG by +6\%. The architecture reduces hallucination by 2.4$\times$ (10\% vs 24\% no-evidence rate). Ablation confirms spirals are critical: single-pass yields 60\%, five spirals reach 74\%, plateauing thereafter. However, VORTEX is sensitive to LLM capability---on weaker models accuracy drops below baseline. We further introduce a Fast/Slow Router that dynamically gates between Naive RAG and VORTEX using the LLM's own uncertainty signal (``Insufficient evidence.''). The router achieves 82\%, outperforming both naive RAG (+14\%) and VORTEX alone (+8\%) with zero additional training. The system is offline-first, config-driven, and validated across three backends (mock, Ollama, OpenAI).
\end{abstract}

\section{Introduction}
\label{sec:intro}

Multi-hop question answering requires retrieving and reasoning over multiple documents. Standard RAG~\cite{lewis2020rag} retrieves once and answers in a single step---it cannot chain evidence across documents.

Recent agentic approaches attempt to address this. A-RAG~\cite{du2026arag} augments RAG with LLM-driven tool selection, but this wastes tokens on routing decisions and introduces failure modes. ReAct~\cite{yao2022react} interleaves reasoning and acting but lacks a convergence signal---it terminates by hard step limit or the LLM's arbitrary decision. RAPTOR~\cite{sarthi2024raptor} builds hierarchical document trees via recursive summarization, requiring expensive index-time preprocessing.

We introduce VORTEX, a system that replaces LLM tool selection with a fixed chained pipeline, replaces arbitrary termination with entropic collapse, and replaces parametric knowledge with fact-free routing. The system is offline-first, config-driven, and validated on CPU hardware.

\section{Related Work}
\label{sec:related}

\textbf{Retrieval-Augmented Generation.} Standard RAG~\cite{lewis2020rag} combines a retriever with a generator. Self-Ask~\cite{press2022selfask} decomposes questions via prompting. Chain-of-Thought~\cite{wei2022cot} uses stepwise reasoning. VORTEX externalizes these steps into explicit retrieval cycles with structured state.

\textbf{Agentic RAG.} ReAct~\cite{yao2022react} alternates reasoning traces with tool calls. A-RAG~\cite{du2026arag} provides hierarchical retrieval interfaces for LLM-driven tool selection. GRASP~\cite{lelong2025grasp} uses graph-based sub-agents with token-efficiency objectives. HERA~\cite{du2026hera} evolves multi-agent orchestration via an experience library. VORTEX differs from all of these by using a fixed chained pipeline (no LLM tool selection), a fact-free planner (zero parametric knowledge), and entropic convergence detection.

\textbf{Hierarchical Retrieval.} RAPTOR~\cite{sarthi2024raptor} builds tree-structured indices via recursive summarization. VORTEX uses flat chunks with iterative spirals---no index-time overhead.

\section{VORTEX Architecture}
\label{sec:architecture}

\subsection{Design Principles}

VORTEX rests on four principles: (1) \textbf{Fact-Free Synapses}---the planner uses 0\% of its capacity for facts and 100\% for routing; (2) \textbf{Entropic Collapse}---convergence is detected via entropy plateau; (3) \textbf{Offline-First}---the default mock mode requires zero dependencies; (4) \textbf{Same Code, Any Hardware}---a config file switches between local CPU (Ollama) and cloud GPU (OpenAI).

\subsection{Gravitational Core (Planner)}

The planner holds zero factual knowledge in its weights. Its state consists of: \texttt{goal\_vector} (immutable original question), \texttt{spiral\_memory} (accumulated \texttt{<fact>} entries), \texttt{hop\_history} (full reasoning log), and confidence/entropy scores tracking convergence. It communicates via an XML contract: \texttt{<step>} emits a sub-question, \texttt{<final\_answer>} emits the answer, and \texttt{<stop\_search>} signals termination.

\subsection{Centrifugal Ingestor (Executor)}

Unlike agentic systems where the LLM selects tools, VORTEX uses a fixed chained pipeline:
\begin{enumerate}
    \item \texttt{keyword\_search(query)}---lexical match returning top-k chunk IDs
    \item \texttt{chunk\_read(id, adjacent=True)}---full text with neighbor chunks
    \item LLM condensation---compresses retrieved text into a structured \texttt{<fact>}
\end{enumerate}
This is cheaper, more predictable, and equally effective for the multi-hop domain.

\subsection{Cyclic Spiral Flow}

Each spiral is one complete cycle: planner emits \texttt{<step>} $\rightarrow$ executor retrieves $\rightarrow$ state updates. Confidence increases monotonically with novel facts. Entropy decreases. Convergence occurs when confidence reaches threshold (0.85), entropy stalls for three consecutive spirals, or safety limits are hit (15 max hops, 4096 token budget).

\subsection{LLM Backends}

Three backends are implemented: MockBackend (canned responses, zero deps for testing), OllamaBackend (local CPU, no API key), and OpenAIBackend (production GPU cluster). Switching is a one-line config change.

\section{Experiments}
\label{sec:experiments}

\subsection{Setup}

We evaluate on a multi-domain corpus (60 chunks, 10 domains) with 50 question-answer pairs. All experiments use Ollama with temperature 0.0 on an NVIDIA T4 GPU (16 GB VRAM) unless noted otherwise. We test four 7-8B models: \texttt{qwen2.5:7b}, \texttt{llama3.1:8b}, \texttt{mistral} (v0.1, 7B), and \texttt{DeepSeek-R1} (7B distill). For each model we run both VORTEX and a Naive RAG baseline (single-pass retrieve + answer, same corpus). Ablation varies \texttt{max\_spirals} from 1 to 15 on \texttt{qwen2.5:7b}.

\subsection{Metrics}

We report: \textbf{Contains} (ground truth is a substring of prediction, main metric), \textbf{No evidence} (fraction of questions where the model admits inability to answer), and \textbf{Spirals} (average cycles per question).

VORTEX achieves its best result on \texttt{qwen2.5:7b}: 74\% vs 68\% baseline ($+6\%$, Fig.~\ref{fig:comparison}). The improvement comes from evidence chaining---VORTEX iteratively retrieves and verifies facts across multiple spirals, while baseline answers from a fixed set of top-3 chunks.

Crucially, VORTEX reduces hallucination: the no-evidence rate drops from 24\% (baseline) to 10\% (VORTEX) on \texttt{qwen2.5:7b}, a 2.4$\times$ improvement. However, on \texttt{llama3.1:8b} and \texttt{mistral}, VORTEX {\it underperforms} baseline. This indicates that VORTEX's spiral architecture requires a minimum LLM reasoning capability---strong models (72\% baseline) or weak models (68\%) both degrade when forced into iterative re-planning.

Single-pass (no spirals) yields 60\% (Fig.~\ref{fig:ablation}, Table~\ref{tab:ablation}). Adding 5 spirals boosts accuracy to 74\%---a +14\% gain. Beyond 5, accuracy plateaus: 10 spirals gives 70\% (within noise) and 15 gives 74\%. This confirms that the spiral mechanism is the primary driver of VORTEX's improvement, and a small depth (5 spirals) is sufficient for convergence on this benchmark.

\subsection{LLM Sensitivity}
\label{sec:llm-sens}

Table~\ref{tab:main} reveals a critical finding: VORTEX {\it helps} \texttt{qwen2.5:7b} (+6\%) but {\it hurts} \texttt{llama3.1:8b} ($-10\%$) and \texttt{mistral} ($-12\%$). The explanation lies in the interaction between the planner's XML protocol and each model's instruction-following profile. \texttt{qwen2.5:7b}---trained on diverse structured data---faithfully executes the XML contract. \texttt{llama3.1:8b}, optimized for conversational chat, struggles with the ``fact-free reasoning axis'' framing, producing degraded outputs in its planner and executor roles. \texttt{mistral} lacks the parametric capacity to handle the multi-step decomposition at all.

A fourth model, \texttt{DeepSeek-R1} (7B distill), exhibits the most extreme sensitivity: it achieves the highest baseline (76\%) but the lowest VORTEX accuracy (57\%), a $-19\%$ degradation. DeepSeek-R1 is trained via reinforcement learning to produce reasoning traces before answering. This reasoning style conflicts with VORTEX's structured XML protocol---the model generates internal chain-of-thought that VORTEX's planner misinterprets as factual content, degrading both planning and condensation. This confirms that VORTEX's architecture favors models optimized for structured instruction following (qwen2.5) over those optimized for chain-of-thought reasoning (DeepSeek-R1).

This establishes a requirement for future work: VORTEX must either adapt its prompts to each LLM family or incorporate a fast-path (System 1) bypass for questions where the model's direct answer is already strong.

\subsection{Fast/Slow Router}

The LLM sensitivity results (Section~\ref{sec:llm-sens}) suggest a dual-process architecture: use VORTEX only when the model's direct answer is uncertain. We implement a Fast/Slow Router on \texttt{qwen2.5:7b}:

\begin{enumerate}
    \item \textbf{Fast path (System 1):} Naive RAG (single-pass retrieve + answer). If the LLM answers confidently (output does not contain ``Insufficient evidence.''), return immediately.
    \item \textbf{Slow path (System 2):} If the LLM says ``Insufficient evidence.'', fall back to full VORTEX spiral engine.
\end{enumerate}

The router uses no training, no confidence threshold, and no separate classifier---the LLM's own uncertainty signal gates the fallback.

\begin{table}[h]
\centering
\begin{tabular}{lcc}
\toprule
Path & Questions & Contains \\
\midrule
Fast (direct answer) & 38/50 (76\%) & 34/38 = 89\% \\
Slow (VORTEX fallback) & 12/50 (24\%) & 7/12 = 58\% \\
\midrule
Total & 50 & \textbf{41/50 = 82\%} \\
\bottomrule
\end{tabular}
\caption{Router breakdown on \texttt{qwen2.5:7b}. 12 questions triggered fallback; VORTEX saved 7.}
\label{tab:router}
\end{table}

The router achieves 82\%---a $+14\%$ improvement over Naive RAG and $+8\%$ over VORTEX alone (Table~\ref{tab:main}). Average query time drops from 59s (VORTEX) to 12s (router), as 76\% of questions take the fast path ($\sim$1s each). This is the key practical result: a zero-training confidence gate matches or exceeds the gains of trained methods like Self-RAG~\cite{asai2024selfrag} and CRAG~\cite{yan2024crag}, which report 5--15\% improvements over baselines at similar model scales.

Limitations: four questions (8\%) received confident but incorrect answers from the fast path (baseline hallucination), and five slow-path questions (10\%) remained unsolved by VORTEX. These together account for the remaining 18\% error rate.

\subsection{Cross-Model Router}

The same-model router uses one LLM for both paths. However, Table~\ref{tab:main} reveals complementary strengths: \texttt{DeepSeek-R1} has the strongest baseline (76\%), \texttt{llama3.1:8b} has the second-strongest (72\%), and \texttt{qwen2.5:7b} excels at VORTEX spirals (74\%). This motivates a cross-model architecture: fast path with a strong-direct-answer model, slow path with \texttt{qwen2.5:7b} VORTEX.

We evaluate two configurations. The first uses \texttt{llama3.1:8b} (fast, 72\% baseline) and \texttt{qwen2.5:7b} VORTEX (slow). The fast path handles 41/50 questions (82\%) at 93\% accuracy; the slow path recovers 4 of 9 fallbacks (44\%). Overall: \textbf{84\%} (42/50)---the best result on our benchmark.

The second configuration uses \texttt{DeepSeek-R1} (fast, 76\% baseline) and \texttt{qwen2.5:7b} VORTEX (slow). Despite DeepSeek's higher baseline, this achieves only 82\% (41/50): the fast path handles 42/50 questions at 90\% accuracy, but the slow path recovers only 3 of 8 fallbacks (38\%). The improvement comes from \texttt{llama3.1:8b}'s more conservative uncertainty estimation---it sends 24\% of questions to fallback vs.\ 16\% for DeepSeek, and VORTEX rescues more of those uncertain cases (58\% vs.\ 38\%). This suggests that fast-model confidence calibration, not raw accuracy, is the key factor in cross-model router design.

\section{Conclusion}
\label{sec:conclusion}

VORTEX demonstrates that a fact-free planner with fixed chained retrieval and entropic collapse is a viable architecture for multi-hop QA. On \texttt{qwen2.5:7b}, VORTEX achieves 74\% Contains (+6\% over Naive RAG) with 2.4$\times$ fewer hallucinations. Ablation confirms that the spiral mechanism is responsible for these gains, with 5 spirals sufficient for convergence.

The Fast/Slow Router extends this to 82\% (same-model) and 84\% (cross-model) by gating between direct answer and spiral retrieval using the LLM's own uncertainty signal. Cross-model routing---fast path with \texttt{llama3.1:8b}, slow path with \texttt{qwen2.5:7b} VORTEX---is the best-performing configuration. We further show that reasoning-optimized models (DeepSeek-R1) achieve the strongest baseline (76\%) but are incompatible with VORTEX's structured protocol (57\%), and that fast-model uncertainty calibration matters more than raw accuracy in cross-model router design.

The system is offline-first, config-driven, and scales from laptop to cluster without code changes. However, VORTEX is not model-agnostic---it requires models optimized for structured instruction following rather than chain-of-thought reasoning, which limits its applicability to certain LLM families.

Several directions emerge from these findings. First, the XML protocol may be the primary barrier for reasoning-optimized models: \texttt{DeepSeek-R1} achieves 76\% baseline (best) but 57\% VORTEX (worst), yet it shares the same base architecture (\texttt{Qwen2.5-7B}) as our best VORTEX model (74\%). If prompt adaptation---replacing XML tags with natural-language instructions---closes this gap, VORTEX becomes model-agnostic, which is the central open question. Second, the router's confidence gate (``Insufficient evidence'' string match) is a heuristic; explicit confidence prompting or temperature-based uncertainty estimation could push accuracy above 85\%. Third, domain-level analysis may reveal where spirals help most---multi-hop questions requiring evidence chaining should benefit more than single-hop lookups. Finally, scaling to HotpotQA~\cite{yang2018hotpotqa} would validate generalizability beyond our 50-question benchmark.
